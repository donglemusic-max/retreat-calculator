# Apps Script — Phase 1: 응답 시트 정리/마이그레이션

기존 구글폼 응답 시트를 **그대로 보존**한 채, 오른쪽에 "앱 컬럼"을 자동으로 채워주는 스크립트입니다.
(가족/그룹 복원 · 금액 계산 · "확인필요" 자동 플래그)

## 배포 방법 (사장님/대표 계정에서 1회)

1. 응답 시트 열기 → 상단 메뉴 **[확장 프로그램] → [Apps Script]**
2. 편집기에 `Migrate.gs` 내용을 통째로 붙여넣고 💾 저장
3. 함수 목록에서 **`enrichSheet`** 선택 → **▶ 실행**
4. 최초 1회 **권한 승인** (이 시트를 편집하도록 허용)
5. 시트 오른쪽 끝에 아래 컬럼이 채워집니다. (다시 실행하면 재계산 = 멱등)

> 폼에 새 응답이 계속 들어오므로, 갱신하려면 `enrichSheet`를 다시 실행하거나,
> `installOnFormSubmit`을 1회 실행해 **폼 제출 시 자동 갱신** 트리거를 걸면 됩니다.

## 추가되는 컬럼

| 컬럼 | 의미 |
|---|---|
| 제출경로 | 기존폼 / (앞으로) 앱 |
| 그룹ID | 이메일 기준 가족·그룹 묶음 ID (G001…) |
| 그룹대표(추정) | 대표자칸 → 입금자명 → 장년부 순으로 추정 |
| 그룹인원(제출) | 같은 그룹으로 실제 제출된 인원 |
| 1인등록비 | 그 사람 부서 등록비 |
| 본인객실 | 개인 객실가(교회배정·부분그룹일 때 1만/4만), 그룹 객실신청이면 0 |
| 본인버스 | 38,000 또는 0 |
| 본인설악산 | 10,000 또는 0 |
| 그룹공동비용(객실+투숙) | 객실·투숙 추가비용 (그룹당 1회, **대표행에만**) |
| 그룹총액 | 그룹 전체 합계 (**대표행에만**) |
| 확인필요 | 명단(7번)에 더 적혀 있는데 제출 안 됨 → `Y` |
| 비고 | 확인필요 사유 + 원본 명단 텍스트 |

> 합계 검산: `Σ(1인등록비+본인객실+본인버스+본인설악산) + Σ그룹공동비용 = 전체 총액` (중복 없음)

# Phase 2: 계산기 → 시트 직접 제출 (`Submit.gs`)

`Submit.gs`를 **같은 Apps Script 프로젝트에 새 파일로 추가**하면 계산기에서 바로 제출 → 시트 적재가 됩니다.
(상수/함수는 `Migrate.gs` 것을 재사용하므로 중복 선언 금지)

## 웹 앱 배포
1. Apps Script 편집기 → **[배포] → [새 배포]**
2. 유형 ⚙️ → **웹 앱**
3. 실행: **나(소유자)** · 액세스: **모든 사용자**
4. 배포 → 나오는 **웹앱 URL**(`...../exec`) 복사
5. 계산기(Vercel)에 연결: 프로젝트 **Settings → Environment Variables → `VITE_SUBMIT_URL`** 에 그 URL 넣고 **Redeploy**
   - (또는 임성현에게 URL 전달 → 코드 상수에 박아 push)

## 제출되는 행
- 폼 응답과 **동일한 컬럼 구조**로 적재 (제출경로=`앱`, 새 그룹ID `A...`).
- 개인=1행, 그룹=구성원 수만큼 N행. 객실/투숙/명단은 그룹 공통, 부서/성별/버스/입금자명은 개인별.
- 앱 컬럼(금액·확인필요)도 함께 채워져 관리자뷰(Phase 3)에서 바로 집계 가능.

# Phase 3a: 본인 조회 / 수정 (같은 `Submit.gs`)

`Submit.gs`의 `doPost`가 `action`으로 분기합니다:
- `submit`(기본): 신규 제출
- `lookup`: `{action:'lookup', name, contact}` → 이름+연락처(숫자만) 일치 행 반환
- `update`: `{action:'update', row, name, contact, fields:{...}}` → 해당 행의 개인 필드(성별/연락처/이메일/캠퍼스/부서/버스/설악산/문의) 수정 + 금액·그룹총액 자동 재계산
  - 낙관적 동시성: row 위치가 밀렸으면 이름+연락처 불일치로 거부 → "다시 조회"

> ⚠️ **재배포 필요**: Submit.gs를 갱신했으면 [배포]→[배포 관리]→기존 배포 ✏️→버전 "새 버전"으로 업데이트하세요.
> (새 배포를 만들면 URL이 바뀌어 계산기 연결이 끊깁니다. 같은 URL 유지하려면 기존 배포의 버전만 올릴 것.)

- 검증(Node 시뮬, 실데이터): lookup 김영표 1건 / recalc 저장값 1,818,000 일치 / 버스 수정 시 +38,000 반영 OK.
- 객실/투숙/그룹 구성은 본인수정 대상 아님(그룹 desync 방지) → 안내데스크/관리자 처리.

# Phase 3b: 관리자 페이지 (같은 `Submit.gs`)

`doPost` action 추가: `admin`(전체조회) / `adminSet`(셀 1개) / `adminBatch`(방배정 일괄). 모두 `pin` 검증.
- **PIN**: `Submit.gs` 상단 `ADMIN_PIN` (기본 `'2026'` → 바꾸세요).
- 관리자 전용 컬럼 `입금확인 / 배정방 / 관리자메모`는 `admin` 호출 시 **없으면 자동 생성**(Migrate 재실행 불필요).
- 접근: 계산기 URL 뒤에 **`#admin`** (예: `.../#admin`) → PIN 입력.
- 기능: 요약 대시보드 / 방배정(자동 제안=캠퍼스·성별·부서 8인 묶음, 편집·저장) / 리마인드(미입금 체크·확인필요) / 캠퍼스별 버스명단.
- 검증(Node 시뮬): 교회배정 31명 → 5개 방 자동 제안 OK.

> ⚠️ **재배포 1회로 3a+3b 동시 반영**: Submit.gs를 최신본으로 교체 후 기존 배포의 버전만 올리면 lookup/update/admin 모두 활성화됩니다.

## 그룹 묶기 = union-find (이메일 ∪ 전화번호 ∪ 대표자G열)

- 같은 이메일/전화면 병합. 추가로 **G열(대표자 이름)을 적은 사람**들은 그 대표자 기준으로 병합(같은 대표자값 + 그 이름의 본인 행).
- 객실/투숙 공동비용을 **그룹당 1회만** 계상 → 그룹원이 서로 다른 이메일로 제출해도 중복 안 됨.
- ⚠️ **입금자명·명단(L열)은 병합에 안 씀**: "한 사람이 적은 주장"이라 동의 안 한 사람이 잘못 묶임(이혜란·예성이 류). 명단은 **확인필요 플래그 + 방배정 자동 페어링**에만 사용.
- 클러스터 안에 "N인 투숙" 행이 있으면 그 방의 그룹가+투숙비 1회, 없으면 각자 개인 객실가.

## 실데이터 검증 결과 (회사 사본, 185명 기준)

- 그룹 **86개** (이메일만 92 → +전화 89 → +대표자G 86)
- **전체 합계 64,616,000원** (인당합+공동합 검산 일치 ✔)
- 과병합 0건 (캠퍼스 다른 조형만 가족 1건은 실제 가족이라 정상)
- **확인필요** 자동 탐지 (명단 인원 > 실제 제출 — "황성수 미제출" 류)

### ⚠️ 객실비용 규칙 (중요 — v2에서 수정)
- **"N인이 투숙"(그룹 객실 신청)**: 그룹가(소노벨스위트 6만 / 소노캄 24만)를 **그룹당 1회** + 투숙비.
- **"교회 배정/부분그룹"(개인)**: 개인 객실가(소노벨스위트 1만 / 소노캄 4만)를 **인당** (`본인객실` 컬럼).
- 초판은 모두 그룹가로 계산해 11그룹 980,000원 과다계상 → 수정함. (재실행 필요)

## 컬럼 매핑 (헤더 텍스트로 자동 탐색 — 순서 바뀌어도 동작)

이메일 / `등록자 이름`(첫번째) / `그룹의 대표자` / `소속부서` / `객실` /
`가족/그룹을 신청` / `명단` / `버스 신청 혹은 자차` / `설악산뷰` / `입금자명`(첫번째)

> ⚠️ `대표자`만으로 찾으면 col3(등록자 이름 설명문에 '대표자' 포함)에 잘못 걸리므로
> `그룹의 대표자`로 매칭한다. (검증 중 발견·수정한 포인트)

## 한계 (의도된 동작)

- 그룹 묶기는 **이메일** 기준. 부부가 다른 이메일로 따로 내면 별도 그룹 → 명단/입금자명으로 `확인필요` 표시.
- 명단은 자유 텍스트라 인원수 추정이 완벽하지 않음 → **숫자는 참고용, 원본 명단을 비고에 그대로** 실어 사람이 검토.
- 이 한계들은 통화에서 합의된 "애매하면 사람이 확인" 방침과 동일.
